#Antibiotic / PatternSpectrum Score
Methods & Calculations

Core Definitions

Spectrum Score — A numerical weight assigned to each antibiotic representing its breadth of antimicrobial spectrum. Scores range from 2 (narrowest) to 22 (broadest). Scores are defined in the Spectrum Scoring tab and can be customized.
Spectrum Score per row = lookup(DrugName) from the scoring table
DOT (Days of Therapy) — Counts one unique antibiotic day per patient-drug-date-provider combination. Duplicate rows (same PatientID, DrugName, AdministrationDate, OrderingPhysicianName) are deduplicated so each combination contributes at most 1 DOT.
DOT = 1 if first occurrence of (PatientID + DrugName + Date + Provider), else 0
Encounter — One unique patient-provider pairing. Defined as the first occurrence of each (PatientID, OrderingPhysicianName) combination in the dataset. A provider who treated the same patient multiple times for different drugs still counts as one encounter with that patient.
Encounter = 1 if first occurrence of (PatientID + OrderingPhysicianName), else 0

Derived Metrics

DASC (Drug-Associated Spectrum Coverage) — Sum of spectrum scores for all systemic antibiotic DOT assigned to a provider. Because DOT is binary (0 or 1), DASC equals the sum of spectrum scores for qualifying rows.
DASC (provider) = Σ SpectrumScore [for all systemic ABX rows where DOT = 1]
DASC/DOT — Average spectrum score per antibiotic day. Represents the intensity of antibiotic coverage relative to the number of unique antibiotic days prescribed.
DASC/DOT = DASC ÷ Sum_DOT
DOT per Encounter — Average antibiotic days prescribed per unique patient encounter with a provider.
DOT/Encounter = Sum_DOT ÷ Sum_UniqueMDEncounter

Filters Applied Before Analysis

Only rows meeting all of the following criteria are included in provider-level calculations:
SystemicABX = 1 DOT = 1 DrugName not in exclusion list Provider encounters ≥ min. threshold
Systemic routes include: IV, IM, PO, oral, intravenous, intramuscular, NG/OG tube, nasogastric, per OG tube, feeding tube, g-tube, intraperitoneal, j-tube.

Outlier Detection

Outlier definition: A provider is classified as an outlier if their (DASC/DOT, DOT/Encounter) coordinates lie outside the selected confidence ellipse. The ellipse is constructed using a covariance-based method equivalent to ggplot2::stat_ellipse() in R, using the Mahalanobis distance with a chi-squared critical value corresponding to the selected confidence level (df = 2).
Chi² critical values (df=2): 80% → 3.219 90% → 4.605 95% → 5.991 (default) 99% → 9.210 Outlier if: Mahalanobis²(provider) > chi²_critical

Data Processing Notes

• All data processing occurs entirely in the browser using JavaScript. No data is ever sent to a server.
• CMI (Case Mix Index) is used for bubble color in scatter plots only; it does not affect DASC or DOT calculations.
• Combined antibiotic names (e.g., Piperacillin-Tazobactam) are matched by substring and must appear in the correct order in the scoring table to take precedence over single-agent matches.
Data Privacy Notice
Fully Client-Side · No Server Transmission
Upload de-identified data only

All data uploaded to this application must be fully de-identified prior to upload. This includes removing or anonymizing all direct identifiers such as patient names, contact information, and any other personally identifiable information (PII).

Dates must be date-shifted before upload

All dates (admission, discharge, administration dates) must be date-shifted prior to upload. Date shifting preserves relative temporal relationships while removing real-world date anchors that could be used to re-identify patients.

Patient IDs must be anonymized but remain unique

The PatientID field must be replaced with a pseudonymous unique identifier (e.g., a hash or sequential number) before upload. IDs must remain unique across the dataset to preserve the integrity of DOT and encounter calculations, but must not be traceable to real patients.

No data is stored, transmitted, or sent externally

This application does not communicate with any backend server, API, or external service. No data from uploaded files is persisted in cookies, localStorage, indexedDB, or any other browser storage mechanism. All data exists only in memory during the current browser session and is irrevocably erased when the page is closed or refreshed.

All processing occurs locally in your browser

Data parsing, spectrum score calculation, provider-level aggregation, statistical analysis, ellipse computation, and all chart rendering are performed entirely using client-side JavaScript running in your browser. External CDN libraries (Plotly, Chart.js, etc.) are loaded from public CDNs for functionality only — no data is passed to these services.

User responsibility: It is the responsibility of the data uploader to ensure all data meets applicable de-identification standards (e.g., HIPAA Safe Harbor or Expert Determination) before use with this tool. This application does not validate or enforce de-identification compliance.
